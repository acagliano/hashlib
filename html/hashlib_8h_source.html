43 Secure Psuedorandom Number Generator (SPRNG)
44 
45 A secure PRNG is an random number generator that produces strings of random bytes
46 such that, given the prior output of the PRNG (bits 0=>i), the next bit (i+1) of
47 the output cannot be predicted by a polynomial time statistical test with a probability
48 non-negligibly greater than 50%.
49 In simpler terms, it produces unpredictable, "entropic" output.
50 When generating values for anything in this library that requires a key or a
51 salt/iv or other random value, USE THIS SPRNG. Do not use the toolchain's random
52 functions for cryptography. They are deterministic (predictable) algorithms.

87 /*
88  SHA-256 Cryptographic Hash
89 
90  A cryptographic hash is used to validate that data is unchanged between two endpoints.
91  It is similar to a checksum, but checksums can be easily fooled; cryptographic hashes
92  are a lot harder to fool due to how they distribute the bits in a data stream.
93  The general use of a hash is as follows: the party sending a message hashes it and
94  includes that hash as part of the message. The recipient hashes the message (except the hash)
95  themselves and then compares that hash to the one included in the message. If the hashes match,
96  the message is complete and unaltered. If the hashes do not match, the message is incomplete
97  or has been tampered with.
98 */

459 /*
460  RSA Public Key Encryption
461 
462  Public key encryption is a form of asymmetric encryption.
463  This means that a key only works in one direction, and both parties need a public key
464  and a private key. The private key is used to decrypt a message and the public key is
465  used to encrypt.
466  In RSA, the public and private keys are modular inverses of each other, such that:
467  encrypted = message ** public exponent % public key, and
468  message = encrypted ** private exponent % private modulus
469  ** means power, % means modulus
470 
471  65537 (and a few other Fermat primes) are commonly used as public exponents.
472  The public key (modulus) is sent in the clear and is known to to everyone.
473  The cryptographic strength of RSA comes from the difficulty of factoring the modulus.
474  Asymmetric encryption is generally VERY slow. Using even RSA-1024 on the TI-84+ CE will
475  take several seconds. For this reason, you usually do not use RSA for sustained encrypted
476  communication. Use RSA to share a symmetric key, and then use AES for future messages.
477 
478  RSA is also involved in SSL signature verification for some certificates that still use
479  that signing algorithm (RSA w/ Sha-256). Most sigs use ECDSA nowadays though, so if you
480  want a certificate that is compatible with this library, specify RSA with SHA-256 as your
481  signature algorithm when creating it.
482  */
