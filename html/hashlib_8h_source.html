43 Secure Psuedorandom Number Generator (SPRNG)
44 
45 A secure PRNG is an random number generator that produces strings of random bytes
46 such that, given the prior output of the PRNG (bits 0=>i), the next bit (i+1) of
47 the output cannot be predicted by a polynomial time statistical test with a probability
48 non-negligibly greater than 50%.
49 In simpler terms, it produces unpredictable, "entropic" output.
50 When generating values for anything in this library that requires a key or a
51 salt/iv or other random value, USE THIS SPRNG. Do not use the toolchain's random
52 functions for cryptography. They are deterministic (predictable) algorithms.

87 /*
88  SHA-256 Cryptographic Hash
89 
90  A cryptographic hash is used to validate that data is unchanged between two endpoints.
91  It is similar to a checksum, but checksums can be easily fooled; cryptographic hashes
92  are a lot harder to fool due to how they distribute the bits in a data stream.
93  The general use of a hash is as follows: the party sending a message hashes it and
94  includes that hash as part of the message. The recipient hashes the message (except the hash)
95  themselves and then compares that hash to the one included in the message. If the hashes match,
96  the message is complete and unaltered. If the hashes do not match, the message is incomplete
97  or has been tampered with.
98 */

457 /*
458  RSA Public Key Encryption
459 
460  Public key encryption is a form of asymmetric encryption.
461  This means that a key only works in one direction, and both parties need a public key
462  and a private key. The private key is used to decrypt a message and the public key is
463  used to encrypt.
464  In RSA, the public and private keys are modular inverses of each other, such that:
465  encrypted = message ** public exponent % public key, and
466  message = encrypted ** private exponent % private modulus
467  ** means power, % means modulus
468 
469  65537 (and a few other Fermat primes) are commonly used as public exponents.
470  The public key (modulus) is sent in the clear and is known to to everyone.
471  The cryptographic strength of RSA comes from the difficulty of factoring the modulus.
472  Asymmetric encryption is generally VERY slow. Using even RSA-1024 on the TI-84+ CE will
473  take several seconds. For this reason, you usually do not use RSA for sustained encrypted
474  communication. Use RSA to share a symmetric key, and then use AES for future messages.
475 
476  RSA is also involved in SSL signature verification for some certificates that still use
477  that signing algorithm (RSA w/ Sha-256). Most sigs use ECDSA nowadays though, so if you
478  want a certificate that is compatible with this library, specify RSA with SHA-256 when creating it.
479  */
